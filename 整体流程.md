# 一、项目整体说明

本项目为基于 OpenGauss 数据库的校园一卡通管理系统，采用前后端分离架构实现。系统以校园卡为核心业务载体，将学生、教师、校园卡、账户、充值、消费、商户、商品、门禁、图书借阅、考勤、通勤车、统计分析和操作日志等功能整合到统一平台中。

系统交付内容包括：

- 前端项目：`campus-card-web`，负责页面展示、路由控制、表单交互和接口调用。
- 后端项目：`campus-card-server`，负责接口封装、业务校验、事务处理、数据访问和缓存维护。
- 数据库脚本：根目录 `init.sql`，作为系统初始化建表、约束、索引和基础字典数据的交付基准。
- 项目文档：接口文档、数据库设计、系统架构图、ER 图和整体流程文档等。

系统总体建设流程如下：

```text
需求分析
↓
功能模块划分
↓
数据库设计与 init.sql 初始化
↓
后端 Controller、Service、Mapper、Entity 开发
↓
前端页面、路由、接口封装与状态管理开发
↓
Redis 缓存接入与热点数据优化
↓
前后端联调与业务闭环测试
↓
系统运行验证
↓
论文和答辩材料整理
```

# 二、技术架构与运行环境

## 2.1 开发环境

| 工具 | 版本或说明 |
|------|------------|
| JDK | 17 |
| Spring Boot | 3.2.x |
| Maven | 后端依赖管理与构建 |
| Node.js | 18 |
| npm | 10 |
| Vue | 3 |
| OpenGauss | 核心业务数据库 |
| Redis | 热点数据缓存 |

## 2.2 技术栈

| 层次 | 技术 | 作用 |
|------|------|------|
| 前端展示层 | Vue 3、Vite、Element Plus | 构建后台管理端和学生、教师自助端页面 |
| 前端状态与路由 | Pinia、Vue Router | 保存登录用户状态，控制不同角色可访问页面 |
| 前端通信 | Axios | 统一封装 HTTP 请求和错误提示 |
| 后端接口层 | Spring Boot Controller | 接收请求、校验基础参数、返回统一响应 |
| 后端业务层 | Service、事务管理 | 实现业务规则、状态流转和数据一致性控制 |
| 数据访问层 | MyBatis-Plus、Mapper | 完成分页、条件查询、增删改查和聚合统计 |
| 数据库层 | OpenGauss | 保存人员、校园卡、账户、交易、考勤等核心数据 |
| 缓存层 | Redis | 缓存校园卡、余额、商户和通勤车基础数据 |
| 文件处理 | EasyExcel、MultipartFile | 支持学生导入导出、图片和申报材料上传 |
| 支付集成 | 支付宝沙箱电脑网站支付 | 实现个人中心在线充值闭环 |

## 2.3 系统分层流程

```text
用户浏览器
↓
Vue 前端页面
↓
Axios 请求封装
↓
Spring Boot Controller
↓
Service 业务处理与事务控制
↓
MyBatis-Plus Mapper
↓
Redis 缓存 / OpenGauss 数据库
```

前端通过 `src/utils/request.js` 统一设置 `/api` 基础路径和响应拦截，后端通过 `Result` 结构统一返回 `code`、`msg`、`data`。业务异常由全局异常处理器统一转为错误响应，前端根据响应结果展示成功或失败提示。

# 三、系统角色与权限流程

## 3.1 角色划分

| 角色 | 登录后入口 | 主要职责 |
|------|------------|----------|
| 管理员 | 首页 | 负责基础数据维护、业务管理、统计查看和系统日志审计 |
| 学生 | 个人中心 | 维护个人资料，查询校园卡，完成消费、借阅、考勤申报、打卡和通勤车登记 |
| 教师 | 个人中心 | 维护个人资料，发布打卡位置，审批负责学生的请假和实习申报 |

## 3.2 登录与权限控制流程

1. 用户在登录页输入账号和密码。
2. 前端调用 `/api/admin/login` 提交登录请求。
3. 后端根据 `admin_user` 表中的用户名和 MD5 密码查询登录账号。
4. 系统读取账号角色：
   - `admin` 或 `SUPER_ADMIN` 归一为管理员。
   - `student` 根据学号查询学生信息。
   - `teacher` 根据教师编号查询教师信息。
5. 若登录用户是学生或教师，后端继续查询其校园卡信息，并返回 `businessUserId`、`cardId`、`cardNo`。
6. 前端将登录用户和本地登录标记保存到 `localStorage`，并由路由守卫控制访问权限。
7. 管理员默认进入首页，学生和教师默认进入个人中心。
8. 非管理员访问管理端页面时，前端提示无权限并跳转到个人中心。

## 3.3 前端菜单流程

管理员菜单覆盖系统全部管理功能：

- 首页数据概览。
- 用户管理：学生管理、教师管理。
- 业务管理：商户、商品、校园卡、账户、充值、消费。
- 系统管理：数据统计、图书、门禁、考勤、通勤车、学院专业、操作日志。

学生菜单主要面向个人自助业务：

- 个人中心：个人资料、校园卡信息、充值、消费记录、图书借阅、借阅记录、考勤申报和打卡。
- 商品消费：浏览商品、选择数量、确认支付。
- 通勤车查询：查询线路、站点、时刻表并完成上车登记。

教师菜单主要面向教学管理和审批：

- 个人中心：个人资料、校园卡信息、消费记录和密码修改。
- 打卡位置管理：创建、编辑、查看本人发布的打卡位置。
- 申报审批：审核自己负责学生的请假和实习申报。
- 通勤车查询：查看线路和班次信息。

# 五、核心业务流程

## 5.1 用户管理流程

### 5.1.1 学生新增流程

1. 管理员进入学生管理页面，填写学号、姓名、性别、学院、专业、班级、手机号、负责老师和初始密码。
2. 前端调用 `/api/student` 新增学生。
3. 后端校验学生基础信息、手机号格式、学号唯一性和登录账号唯一性。
4. 学生信息写入 `student` 表。
5. 系统同步创建 `admin_user` 登录账号，角色为 `student`，密码使用 MD5 加密保存。
6. 学生可使用学号和密码登录系统。

### 5.1.2 学生批量导入流程

1. 管理员下载学生导入模板。
2. 按模板填写学号、姓名、学院、专业、班级。
3. 前端上传 Excel 文件到 `/api/student/import`。
4. 后端逐行读取并校验数据。
5. 对已存在学号进行跳过，对缺失字段或登录账号冲突的数据记录失败原因。
6. 成功导入的学生默认创建登录账号，初始密码为 `123456`。
7. 返回总数、成功数、跳过数、失败数和失败原因，便于管理员修正数据。

### 5.1.3 学生资料完整性控制

学生开卡时不强制校验资料完整性，但在实际使用学生端关键业务时会再次校验资料是否完整。资料完整要求包括姓名、性别、学院、专业、班级和手机号，手机号还必须符合格式要求。

资料不完整时，学生不能进行以下操作：

- 商品消费。
- 图书借阅。
- 通勤车上车登记。
- 考勤申报。
- 考勤打卡。

这种设计既允许管理员先批量导入基础学生数据，又能保证实际业务发生时学生身份资料可追溯。

### 5.1.4 教师管理流程

教师管理流程与学生类似。管理员维护教师编号、姓名、性别、部门、手机号和初始密码，后端写入 `teacher` 表，并同步创建角色为 `teacher` 的登录账号。教师登录后可在个人中心维护个人信息和修改密码。

## 5.2 校园卡与账户流程

### 5.2.1 开卡流程

1. 管理员进入校园卡管理页面。
2. 选择学生或教师，也可以筛选“未开卡”人员后批量勾选。
3. 前端调用 `/api/card/open` 或 `/api/card/batch-open`。
4. 后端校验用户类型和用户编号，确认学生或教师存在。
5. 系统检查该用户是否已有在用校园卡，未注销的卡不允许重复开卡。
6. 后端生成唯一卡号，写入 `campus_card` 表。
7. 系统同步创建账户，写入 `account` 表，初始余额为 0。
8. 系统记录开卡变更信息，便于后续追溯。

### 5.2.2 挂失流程

1. 管理员选择正常状态校园卡并填写挂失原因。
2. 前端调用 `/api/card/loss`。
3. 后端校验校园卡存在且状态为正常。
4. 更新 `campus_card.status` 为挂失。
5. 同步更新对应 `account.status`。
6. 写入 `card_change_record` 操作记录。
7. 删除校园卡信息和余额相关 Redis 缓存。

### 5.2.3 解挂流程

1. 管理员选择挂失状态校园卡并填写解挂原因。
2. 前端调用 `/api/card/unloss`。
3. 后端校验校园卡存在且状态为挂失。
4. 更新校园卡状态为正常，并同步恢复账户状态。
5. 写入解挂操作记录。
6. 清除相关缓存。

### 5.2.4 注销流程

1. 管理员选择校园卡并填写注销原因。
2. 前端调用 `/api/card/cancel`。
3. 后端校验校园卡存在。
4. 如果校园卡仍在有效期内，必须填写注销原因，例如转学、辍学、离职等。
5. 更新校园卡状态为注销，并同步更新账户状态。
6. 写入注销操作记录。
7. 清除校园卡和余额缓存。

## 5.3 充值流程

### 5.3.1 后台充值流程

1. 管理员进入充值管理页面，输入校园卡或卡号、充值金额和充值方式。
2. 前端调用 `/api/recharge` 或 `/api/recharge/by-card-no`。
3. 后端校验充值金额必须大于 0，充值方式不能为空。
4. 系统校验校园卡存在，且不是注销或挂失状态。
5. 查询校园卡对应账户。
6. 更新账户余额。
7. 写入 `recharge_record` 充值记录。
8. 写入 `account_flow` 账户流水，变动类型为“充值”。
9. 删除 `account:balance:{cardId}` 余额缓存。

### 5.3.2 支付宝沙箱充值流程

1. 学生或教师在个人中心选择充值金额。
2. 前端调用 `/api/recharge/alipay/page-pay` 创建支付订单。
3. 后端校验校园卡、账户和充值金额。
4. 系统生成本地订单号，写入 `recharge_order` 表。
5. 后端调用支付宝沙箱接口生成电脑网站支付表单。
6. 前端跳转到支付宝沙箱收银台完成支付。
7. 支付完成后，前端调用 `/api/recharge/alipay/status/{outTradeNo}` 主动查询订单状态。
8. 若支付宝确认支付成功，后端更新本地订单状态。
9. 系统完成入账，写入充值记录和账户流水，并清除余额缓存。

该流程通过本地订单号唯一约束保证支付订单可追踪，并通过主动查询支付状态完成支付闭环。

## 5.4 消费流程

### 5.4.1 管理端消费流程

1. 管理员在消费管理中录入校园卡、商户和金额。
2. 前端调用 `/api/consume` 或 `/api/consume/by-card-no`。
3. 后端校验消费金额大于 0。
4. 校验商户存在且状态正常。
5. 校验校园卡存在，且不是注销或挂失状态。
6. 若校园卡属于学生，校验学生资料完整性。
7. 查询账户并校验账户状态正常。
8. 使用余额条件更新扣款，避免余额不足时出现并发扣减问题。
9. 写入 `consume_record` 消费记录。
10. 写入 `account_flow` 账户流水，变动类型为“消费”。
11. 删除余额缓存。

### 5.4.2 商品消费流程

1. 学生进入商品消费页面。
2. 页面加载商品列表、商户列表和当前校园卡余额。
3. 学生选择商品和数量后确认支付。
4. 前端调用 `/api/consume/product-order`。
5. 后端校验商品存在、已上架、库存充足。
6. 系统先使用条件更新扣减库存。
7. 后续进入统一消费逻辑，完成校园卡、账户、余额和资料完整性校验。
8. 系统扣减余额，写入消费记录和账户流水。
9. 前端刷新余额和商品库存。

## 5.5 商户与商品流程

1. 管理员维护商户类型，系统通过 `merchant_type` 保存分类。
2. 管理员新增商户，填写商户名称、类型、位置、联系方式、状态和 Logo。
3. 后端校验商户名称和商户类型，并维护商户缓存。
4. 管理员在商品管理中新增商品，填写商品名称、所属商户、价格、库存、状态、描述和图片。
5. 后端校验商户存在且状态正常，商品价格必须大于 0，库存不能小于 0。
6. 商品上架后，学生可在商品消费页面购买。

## 5.6 图书借阅流程

### 5.6.1 图书维护流程

1. 管理员维护图书名称、作者、馆藏地、封面和状态。
2. 图书信息保存到 `book` 表。
3. 图书状态为可借阅时，学生端可以提交借阅申请。

### 5.6.2 借阅申请流程

1. 学生进入个人中心的图书借阅页。
2. 系统加载图书列表、当前借阅数量和禁借状态。
3. 学生点击借阅，前端调用 `/api/borrow/application`。
4. 后端校验校园卡状态正常，学生资料完整。
5. 系统检查是否存在当前逾期或禁借期。
6. 系统计算当前已借阅和待审批数量，总数达到 3 本时禁止继续申请。
7. 校验图书存在且可借阅。
8. 校验同一本书是否已有待审批申请。
9. 写入 `borrow_application`，状态为待审批。

### 5.6.3 借阅审批与归还流程

1. 管理员在借阅申请页面查看待审批记录。
2. 审批通过时，系统再次校验借阅资格和图书状态。
3. 系统写入 `borrow_record` 借阅记录，设置借阅时间、应还时间和借阅中状态。
4. 更新图书状态为已借出。
5. 学生归还图书时，前端调用 `/api/borrow/return`。
6. 后端更新借阅记录为已归还，写入归还时间。
7. 更新图书状态为可借阅。

### 5.6.4 逾期与禁借流程

1. 系统查询借阅记录时，会根据应还时间判断是否逾期。
2. 借阅中且超过应还时间的记录会标记为超期。
3. 若学生存在当前未归还逾期图书，则禁止继续借阅。
4. 若学生曾逾期归还，则按逾期天数计算禁借期。
5. 禁借期内学生不能提交新的借阅申请。

## 5.7 考勤管理流程

### 5.7.1 打卡位置发布流程

1. 教师进入打卡位置管理页面。
2. 填写位置名称、经纬度、允许半径、开始时间、结束时间和状态。
3. 前端调用 `/api/attendance/location`。
4. 后端校验位置名称、经纬度、半径和时间范围。
5. 打卡开始时间和结束时间必须在同一天，结束时间必须晚于开始时间。
6. 保存到 `attendance_location` 表，并绑定发布教师。
7. 教师端只展示自己发布的未删除位置，管理员端可以查看全部位置。

### 5.7.2 学生打卡流程

1. 学生进入个人中心考勤打卡区域。
2. 前端携带校园卡 ID 查询有效打卡位置。
3. 后端根据校园卡找到学生，再根据 `student.teacher_id` 返回负责老师发布、启用且当前有效的位置。
4. 学生提交打卡时，后端校验校园卡状态正常和学生资料完整。
5. 系统再次校验打卡位置属于学生负责老师，防止绕过前端打其他老师的位置。
6. 在开始时间后 15 分钟内打卡记为正常。
7. 超过 15 分钟且未到结束时间打卡记为迟到。
8. 系统写入 `attendance_record`。

### 5.7.3 请假申报流程

1. 学生提交请假申请，填写开始日期、结束日期和请假原因。
2. 线上请假支持 1 到 3 天，超过 3 天需走线下流程。
3. 后端校验学生资料完整，且不存在时间重叠的待审核或已通过请假申请。
4. 申请保存到 `attendance_application`，类型为 `LEAVE`，状态为 `PENDING`。
5. 负责老师在申报审批页面查看申请。
6. 老师审批时，后端校验该学生的 `teacher_id` 是否等于当前老师 ID。
7. 审批通过后，如果请假日期覆盖当天，学生考勤状态更新为请假。
8. 请假有效期结束后，定时任务或查询刷新会恢复学生在校状态。

### 5.7.4 实习申报流程

1. 学生提交实习申报，填写实习单位、开始日期和申报说明。
2. 可通过材料上传接口上传图片、PDF、Excel、Word 等资料，后端限制文件大小和类型。
3. 后端校验学生资料完整，且不存在未结束的待审核或已通过实习申报。
4. 申请保存为 `INTERNSHIP` 类型。
5. 负责老师审批通过后，学生考勤模式更新为实习，考勤状态更新为实习。
6. 实习期间学生支持校外位置打卡或实习日志上报。
7. 实习返校时，老师或管理员调用返校接口，系统将学生考勤模式恢复为在校。

### 5.7.5 缺勤补齐流程

1. 系统定时扫描当天已经结束的打卡位置。
2. 找到负责老师名下应该参与该位置打卡的在校学生。
3. 对未在该位置产生打卡记录的学生生成缺勤记录。
4. 已请假、外出实习、无有效校园卡或资料不完整的学生不生成缺勤。
5. 管理员可在考勤记录和统计页面查看正常、迟到、早退、缺勤等数据。

## 5.8 通勤车流程

### 5.8.1 基础数据维护流程

1. 管理员维护站点，包含站点名称、位置、经纬度和状态。
2. 管理员维护车辆，包含车牌号、车型、座位数和状态。
3. 管理员维护线路，包含线路名称、起点站、终点站、总距离、预计时间和状态。
4. 管理员维护班次，绑定线路、车辆、发车时间、频率、开始日期和结束日期。
5. 系统将线路、车辆和班次基础数据写入数据库，并在查询时使用 Redis 缓存提升读取效率。

### 5.8.2 学生查询流程

1. 学生进入通勤车查询页面。
2. 前端加载线路、车辆、站点和班次信息。
3. 学生授权浏览器定位后，前端计算当前位置到线路起点和终点的距离及步行预计时间。
4. 学生可查看线路站点和指定线路的时刻表。

### 5.8.3 上车登记流程

1. 学生选择线路、车辆和班次。
2. 点击上车后，前端调用 `/api/commute/boarding`。
3. 后端校验校园卡存在且状态正常。
4. 若校园卡属于学生，校验学生资料完整。
5. 校验线路存在且启用。
6. 校验车辆存在且可用。
7. 校验班次存在且启用，并确认班次属于所选线路、车辆属于所选班次。
8. 校验当前日期在班次有效日期范围内。
9. 校验该校园卡当天未登记过同一班次。
10. 系统从 1 号座位开始查找当天该班次未占用座位，自动分配座位号。
11. 如果车辆满员，则拒绝登记。
12. 写入 `commute_record`，并返回记录 ID、线路、车辆、班次、座位号和登记时间。

## 5.9 门禁管理流程

### 5.9.1 门禁点维护流程

1. 管理员维护门禁点名称、位置、设备 ID、经纬度、允许通行半径和状态。
2. 后端校验经纬度范围和通行半径。
3. 门禁点保存到 `access_point` 表。

### 5.9.2 二维码开门流程

1. 用户在门禁页面或扫码入口提交校园卡、门禁点和当前位置。
2. 前端调用 `/api/access/qr`。
3. 后端校验门禁点存在且启用。
4. 后端校验浏览器已提供当前位置。
5. 系统计算当前位置与门禁点坐标的距离。
6. 距离超出门禁点允许半径时拒绝通行。
7. 查询该校园卡在该门禁点的最后一次记录。
8. 如果上一次为“进”，本次自动记录为“出”；否则记录为“进”。
9. 写入 `access_record`，状态为成功。

## 5.10 统计与日志流程

### 5.10.1 首页仪表盘

管理员首页通过 `/api/dashboard` 查看基础运营数据，包括学生数量、教师数量、商户数量和今日消费总额。

### 5.10.2 综合统计

统计页面通过 `/api/stat` 系列接口展示：

- 综合概览：学生、教师、商户、商品、图书、校园卡、充值、消费、借阅、考勤、门禁和通勤数据。
- 消费趋势：按日期统计消费金额。
- 用户排行：统计用户消费排行。
- 商户排行：统计商户收入排行。
- 图书统计：统计当前逾期、已借出、可借出和禁借剩余天数排行。
- 校园卡注销统计：按注销操作记录统计注销原因和明细。
- 考勤趋势：统计正常、迟到、早退和缺勤数量。
- 通勤车统计：保留业务接口，用于后续扩展。

### 5.10.3 操作日志

系统通过 `operation_log` 表保存关键操作。日志查询支持按操作人、操作类型和目标表筛选，导出功能便于审计和论文测试截图整理。

## 6.2 校园卡核心数据关系

校园卡是系统中连接人员和业务的核心对象。学生或教师拥有校园卡，校园卡关联账户，账户产生充值、消费和流水记录。校园卡还会参与门禁、图书借阅、考勤和通勤车业务。

```text
student / teacher
↓
campus_card
↓
account
↓
recharge_record / consume_record / account_flow
```

辅助业务围绕校园卡扩展：

```text
campus_card
├─ access_record
├─ borrow_record
├─ attendance_record
└─ commute_record
```

## 6.3 数据一致性策略

- 业务表普遍使用 `is_deleted` 字段实现逻辑删除。
- 学号、教师编号、管理员用户名、卡号、商户名称、商户类型、学院专业组合和支付宝订单号通过唯一约束或唯一索引控制重复数据。
- 校园卡状态变更时，同步更新账户状态。
- 充值和消费通过事务保证余额、记录和流水一致。
- 商品消费先校验并扣减库存，再进入统一扣款流程。
- 通勤车上车登记通过重复登记校验和座位占用校验保证同一班次数据一致。
- 考勤审批通过后同步更新学生考勤模式和考勤状态。

# 七、数据库设计与 OpenGauss 实现

## 7.1 初始化脚本

系统以根目录 `init.sql` 作为数据库初始化交付脚本。该脚本负责：

1. 创建 `campus_card` schema。
2. 创建学生、教师、学院专业、管理员、校园卡、账户、流水、充值订单、消费、商户、商品、门禁、图书、考勤、通勤车和系统日志等表。
3. 为表和字段添加注释，便于数据库维护和论文说明。
4. 创建唯一约束和唯一索引，避免重复业务数据。
5. 创建高频查询索引，提高列表查询、时间范围查询、状态筛选和统计聚合效率。
6. 插入学院专业基础字典。
7. 插入通勤车站点、车辆、线路和班次初始化数据，便于功能联调。

## 7.2 逻辑外键设计

系统不依赖数据库物理外键，而是在业务层通过字段维护关联关系。例如：

- `campus_card.user_id` 与 `user_type` 关联学生或教师。
- `account.card_id` 关联校园卡。
- `recharge_record.account_id` 关联账户。
- `consume_record.account_id` 和 `merchant_id` 关联账户与商户。
- `borrow_record.card_id` 和 `book_id` 关联校园卡与图书。
- `attendance_record.card_id` 和 `location_id` 关联校园卡与打卡位置。
- `commute_record.route_id`、`vehicle_id`、`schedule_id` 关联通勤车基础数据。

这种设计减少物理外键带来的耦合，便于毕业设计系统扩展和脚本维护，但要求 Service 层严格执行存在性校验和状态校验。

## 7.3 索引设计

系统索引主要覆盖以下场景：

- 人员查询：学号、教师编号、负责老师、考勤模式。
- 校园卡查询：卡号、用户 ID、卡状态。
- 账户查询：卡 ID、账户状态。
- 充值和消费查询：账户 ID、商户 ID、商品 ID、时间和状态。
- 门禁查询：校园卡、门禁点、通行时间和状态。
- 图书查询：校园卡、图书、借阅状态和应还时间。
- 考勤查询：打卡位置、校园卡、考勤时间、考勤类型。
- 通勤车查询：线路、车辆、班次、乘车时间。
- 日志审计：操作人、操作类型、操作时间。

索引设计服务于后台分页查询、统计聚合和论文中的数据库性能优化说明。

# 八、Redis 缓存设计

## 8.1 缓存使用场景

Redis 主要缓存访问频率较高、更新频率相对较低的数据：

| 缓存键 | 内容 | 过期时间 |
|--------|------|----------|
| `card:info:{cardId}` | 按 ID 缓存校园卡信息 | 1800 秒 |
| `card:info:no:{cardNo}` | 按卡号缓存校园卡信息 | 1800 秒 |
| `account:balance:{cardId}` | 缓存账户余额 | 1800 秒 |
| `merchant:info:{merchantId}` | 缓存商户信息 | 3600 秒 |
| `merchant:type:list` | 缓存商户类型列表 | 3600 秒 |
| `commute:route:{routeId}` | 缓存通勤线路信息 | 3600 秒 |
| `commute:route:list` | 缓存线路列表 | 3600 秒 |
| `commute:vehicle:{vehicleId}` | 缓存车辆信息 | 3600 秒 |
| `commute:vehicle:list` | 缓存车辆列表 | 3600 秒 |
| `commute:schedule:{scheduleId}` | 缓存班次信息 | 3600 秒 |
| `commute:schedule:list` | 缓存班次列表 | 3600 秒 |
| `commute:schedule:route:{routeId}` | 缓存指定线路班次 | 3600 秒 |

## 8.2 缓存读写策略

1. 查询数据时，优先读取 Redis。
2. 缓存未命中时查询 OpenGauss。
3. 查询成功后将结果写入 Redis，并设置过期时间。
4. 更新、删除或状态变更后，删除相关缓存。
5. 下次查询时重新从数据库加载最新数据。

## 8.3 缓存与业务一致性

- 校园卡挂失、解挂、注销后删除校园卡信息和余额缓存。
- 充值成功后删除余额缓存。
- 消费成功后删除余额缓存。
- 商户和商户类型新增、修改、删除后删除相关缓存。
- 通勤车线路、车辆和班次变更后删除相关缓存。

这种缓存策略实现简单，适合毕业设计项目，也能说明 Redis 在热点数据读取优化中的作用。

# 九、前后端联动流程

## 9.1 前端请求流程

1. 页面组件触发表单提交或列表查询。
2. `src/api/` 中的模块化接口函数封装请求路径和参数。
3. `src/utils/request.js` 统一添加 `/api` 前缀；若本地存在登录标记，则附加 `Authorization` 请求头。
4. 后端返回统一 `Result`。
5. 响应拦截器判断 `code`。
6. 成功时返回 `data`，失败时显示错误提示。
7. 如果返回 401，前端清除登录状态并跳转登录页。

## 9.2 路由权限流程

1. 路由配置使用 `requiresAuth`、`requiresAdmin` 和 `allowedRoles` 标记权限。
2. 路由守卫从 `localStorage` 读取本地登录标记和用户角色。
3. 未登录访问受保护页面时跳转登录页。
4. 已登录访问登录页时，按角色跳转默认页面。
5. 非管理员访问管理员页面时跳转个人中心。
6. 角色不在允许列表中时提示无权限。

## 9.3 表单与分页流程

大部分管理页使用统一的交互模式：

1. 页面加载时请求列表数据。
2. 查询条件变化后重置到第一页。
3. 分页参数随请求提交到后端。
4. 后端使用 MyBatis-Plus `Page` 对象分页查询。
5. 前端根据 `records` 和 `total` 渲染表格和分页器。
6. 新增、修改、删除成功后刷新列表。

| 挂失 | 校园卡和账户状态同步变更 |
| 充值 | 账户余额增加，充值记录和账户流水同时生成 |
| 消费 | 账户余额减少，消费记录和账户流水同时生成 |
| 商品购买 | 商品库存减少，余额扣减，消费记录保存 |
| 借阅审批 | 审批通过后生成借阅记录，图书状态变为已借出 |
| 图书归还 | 借阅记录写入归还时间，图书状态恢复可借阅 |
| 实习审批 | 学生考勤模式和考勤状态同步更新 |
| 通勤登记 | 乘车记录生成，座位号不重复 |

## 11.2 系统特色说明

本系统的设计重点体现在以下方面：

- 以校园卡为统一身份和业务凭证，连接账户、消费、门禁、图书、考勤和通勤车场景。
- 使用 OpenGauss 保存结构化业务数据，并通过注释、索引和唯一约束提升可维护性。
- 使用 Redis 缓存余额、校园卡、商户和通勤车基础数据，提高热点查询效率。
- 使用前后端分离架构，前端负责角色菜单和交互体验，后端集中处理业务规则。
- 使用 Service 层事务保证充值、消费、借阅审批等关键流程的数据一致性。
- 将学生资料完整性、负责老师审批归属、考勤状态流转、图书逾期禁借等规则落到业务代码中，使系统更接近真实校园业务。

## 11.3 后续可扩展方向

- 增加移动端扫码消费和扫码门禁入口。
- 引入后端签发的 JWT 鉴权和更完整的接口权限拦截。
- 增加商户端角色，使商户可独立查看商品和收入。
- 扩展统计图表维度，例如按学院、班级、商户类型、时段统计。
- 对充值和消费记录进行归档或分区，提高长期运行后的查询性能。
- 对通知场景进行扩展，例如请假审批结果通知、图书到期提醒、校园卡余额提醒。
